# FindJobs 后续实施计划

## 目标与边界

本轮目标是把现有采集 MVP 提升为可用于真实求职决策的本地工具，优先解决岗位误收、岗位状态失真和简历推荐证据不足，再继续扩展公司来源。

必须保持以下边界：

- 只使用公司官网或官网明确链接的 ATS，不采集第三方招聘平台。
- 不采集华为及其关联公司。
- AI 岗位排除纯算法岗，安全算法岗同样排除。
- AI 范围聚焦应用工程、Agent、平台、MLOps、推理部署和 AI 安全。
- Security 范围聚焦安全工程、攻防、漏洞、AppSec、SDL、云安全、数据安全、隐私工程和安全平台；排除普通风控策略、经营分析、产品运营等职能岗。
- 薪资只保存官网明确披露内容；未披露时保持未知，不估算。
- AI 工作流只消费导出事实并输出分析，不直接改写岗位事实或数据库。
- 第一版继续服务个人本地使用，不增加账号系统、云部署和自动投递。

## 执行协议

- Codex 负责设计、任务拆分、数据规则判定、diff 审查、测试、Live smoke 和最终验收。
- 开发执行默认交给 Claude Code + `deepseek-v4-flash[1M]`，每次只执行一个边界清晰的小任务。
- Windows Claude Code 编辑仍未证明可靠时，使用已验证的 WSL Claude Code 显式路径 `/home/zhde/.npm-packages/bin/claude`；执行前检查版本、Base URL、API Key 环境变量和模型配置与 Windows 一致。
- 不使用 `deepseek-v4-pro`，除非用户另行批准。复杂任务由 Codex 继续拆小后交给 flash。
- 同一个失败原因最多尝试 3 次；之后停止，记录命令、错误、已尝试方案和替代建议。
- 核心共享模块串行开发。公司 adapter 可以并行 2 至 3 个任务，但每个任务只能拥有一家公司的 adapter、fixture 和测试；`config/sources.yaml`、注册表和共享分类规则由 Codex 集成。
- 每个任务包必须先补失败测试，再做最小实现；Codex 验收时运行目标测试、全量测试、Ruff 和必要的真实官网 smoke。

## 关键设计决定

### 岗位状态

保留现有 `Job.status` 字段作为官网可用状态，值扩展为：

- `active`：最新可信采集仍存在。
- `missing`：一次可信采集未出现，暂不判断下线。
- `archived`：连续两次可信采集未出现。

新增独立字段：

- `relevance_status`：`target`、`review`、`excluded`。
- `missing_run_count`：连续未出现次数。
- `classification_version`：生成当前分类结果的规则版本。
- `classification_reasons`：JSON 格式的可解释原因码。

只有声明为 `complete_for_target_scope` 的 adapter 才参与 missing/archived 流转。失败、部分采集和未知完整度的采集不能改变未出现岗位的状态。岗位重新出现时恢复 `active` 并清零计数。

### 非破坏性重分类

现有 `prune` 不再删除岗位、观察记录和用户标记。重分类只更新 `relevance_status`、标签、分类版本和原因。旧的硬删除行为改为显式维护命令，第一轮不提供自动调用入口。

### 推荐事实

推荐流程需要使用官网岗位职责和岗位要求。保留当前 `description` 兼容字段，同时为规范化岗位增加 `responsibilities` 和 `requirements` 字段。Adapter 能直接提供两个字段时分别保存；只能提供组合文本时保存原文并使用可测试的分段器拆分，不推断缺失内容。

### 推荐方式

推荐使用确定性基线：硬排除条件、岗位领域、技能覆盖、要求覆盖、经验和地域组成可解释分数。AI 仅对 `review` 队列或确定性结果生成补充分析，并输出结构化文件，不写数据库。

## 阶段 1: 数据状态与迁移基础

**目标**: 建立可升级、非破坏的数据库状态模型，并修复采集运行生命周期。

**任务**:

1. 引入 Alembic，生成当前基线和增量迁移；迁移前自动创建带时间戳的 SQLite 备份。
2. 为 `jobs` 增加 `relevance_status`、`missing_run_count`、`classification_version`、`classification_reasons`、`responsibilities`、`requirements`。
3. 为 `UserMark(job_id, mark_type)` 增加唯一约束，并迁移现有重复数据；保留现有收藏和备注。
4. 为来源配置增加 `collection_completeness`，默认 `partial`，只有验收过的来源可设置为 `complete_for_target_scope`。
5. 在网络采集前创建并提交 `running` 记录；成功和失败都更新同一条 `CollectRun`，不再回滚后新建失败记录。
6. 成功的可信采集完成后执行状态对账；失败或部分采集不增加 `missing_run_count`。
7. 将 `prune` 改为非破坏性重分类；提供 `--dry-run` 和明确的 `--apply`。

**成功标准**:

- 现有 `data/findjobs.db` 可迁移且自动备份，2 个现有 bookmark 保留。
- 当前 2,994 个岗位迁移后仍可读取，不发生岗位、观察记录和用户标记丢失。
- 完整来源中岗位连续缺失一次变为 `missing`，连续缺失两次变为 `archived`，重新出现恢复 `active`。
- 失败采集不会使任何岗位进入 `missing` 或 `archived`。
- `prune --dry-run` 不写数据库，`prune --apply` 不硬删除岗位和标记。

**测试**:

- Alembic 从空库和现有 schema 升级测试。
- 迁移前备份、重复标记合并、标记保留测试。
- 完整采集、部分采集、失败采集、连续缺失、重新出现状态机测试。
- CollectRun 在网络异常、解析异常和数据库异常下的单记录生命周期测试。
- 非破坏性重分类与事务回滚测试。

**状态**: 已完成

## 阶段 2: 岗位相关性高精度修复

**目标**: 消除已知 AI/Security 误收，并让每个分类结果可解释、可回归。

**任务**:

1. 从当前数据库建立真实岗位黄金样本，至少包含 100 个正样本、100 个负样本和 100 个模糊样本。
2. 将已知误收加入负样本，包括普通策划、DBA、车机架构、普通风控产品、风控策略运营、广告运营和游戏安全策划。
3. 将真实 AI 工程、AI 平台、Agent、MLOps、推理部署、AI 安全、AppSec、SDL、攻防、数据安全工程加入正样本。
4. 重构分类器为硬排除、标题/类型强信号、职责工程信号、描述补充信号四层规则。
5. 产品、运营、策略、分析、策划和职能岗默认排除；只有明确研发、工程、架构、平台、检测、响应、攻防等职责才允许安全工程覆盖。
6. 保持标题或岗位类型包含“算法”时直接排除，包括安全算法。
7. 输出原因码，例如 `ai_surface_agent`、`security_appsec`、`excluded_functional_risk`、`excluded_algorithm`。
8. 新增 `findjobs relevance-audit`，输出标签分布、描述单独提升数量、可疑职能岗位、算法残留和随机审计样本。
9. 为模糊岗位增加只读 AI/Skill 审核导出；其结果进入报告，不直接改数据库。
10. 对现有数据库先运行 dry-run 差异报告，由 Codex 审查后再 apply。

**成功标准**:

- 所有已确认误收样本分类为 `excluded`。
- 所有已确认目标样本分类为 `target`。
- 算法及安全算法残留为 0。
- 分类变化不会删除岗位、观察记录或用户标记。
- Codex 人工复核更新后 Top 100 AI、Top 100 Security 和全部 AI Security 岗位，不再发现已知类型的职能误收。

**测试**:

- 黄金样本参数化单元测试。
- 职能风险岗与安全工程覆盖的边界测试。
- 只在岗位要求出现 AI/安全偏好时不得提升的测试。
- 分类原因、版本升级和 dry-run/apply 测试。
- 当前数据库审计回归：算法残留 0、重复身份 0、已知误收 0。

**状态**: 进行中

## 阶段 3: 简历驱动推荐闭环

**目标**: 使用本地简历和官网岗位要求生成可解释、可操作的岗位推荐与学习建议。

**任务**:

1. 实现 `findjobs profile import <docx|pdf>`，使用结构化文档解析库提取文本，不使用正则直接解析 DOCX 二进制。
2. 输出本地忽略的 `profile/profile.json` 和可编辑的 `profile/profile.md`；默认移除姓名、电话、邮箱等联系信息。
3. 画像字段包含技能、经验年限、项目方向、教育、目标城市、目标岗位、排除公司、工作类型和硬约束。
4. 扩展 canonical job 与 adapter，分别保存职责和要求；不能拆分时保留原文并标记 `detail_completeness`。
5. 为导出增加 `--detail-level summary|full`；周报可使用 summary，推荐固定使用 full。
6. 实现 `findjobs recommend`，先应用公司、算法岗、岗位领域等硬过滤，再按技能、职责、要求、经验、地域计算可解释分数。
7. 薪资未披露时不加分、不扣分；不把市场薪资估算混入岗位事实。
8. 生成岗位推荐、匹配证据、缺口、投递建议和学习建议；每条证据可追溯到官网字段或个人画像字段。
9. 增加 `/recommendations` 页面，并复用收藏、已投递、忽略操作。
10. AI 工作流只读取 full 导出和结构化画像，为确定性推荐补充文字建议。

**成功标准**:

- 本地 DOCX 和 PDF 都可导入，联系信息不会出现在导出报告或 Git 跟踪文件中。
- 推荐分数至少包含岗位领域、技能、岗位要求、经验和地域五类可解释证据。
- Top 50 推荐中没有阶段 2 已排除的岗位。
- 每个推荐岗位都包含官网 URL、职责/要求完整度和明确匹配理由。
- 现有简历能生成推荐页、投递优先级和学习建议；不编造薪资或缺失岗位要求。

**测试**:

- DOCX/PDF fixture 解析、隐私字段移除和画像 schema 测试。
- 职责/要求分段和不可分段回退测试。
- 硬排除、经验不足、技能命中、地域命中和薪资未知评分测试。
- 推荐结果稳定排序、证据追踪和 AI 只读边界测试。
- Web 推荐页筛选和用户标记集成测试。

**状态**: 未开始

## 阶段 4: Web 性能与本地运行可靠性

**目标**: 让 3,000 至 10,000 个岗位下的日常筛选、标记和每周运行保持快速、清晰、可恢复。

**任务**:

1. 为岗位列表增加服务端分页、总数、每页数量和稳定排序，默认只显示 `active + target`。
2. 地域和岗位类型继续使用现有规范化逻辑；先在查询后对 ID 集合分页，只有性能分析证明必要时才增加 facet 表，避免过早扩展 schema。
3. 增加中文界面、推荐入口、状态筛选、相关性筛选和默认隐藏 ignored。
4. 支持删除标记；bookmark 可与 applied 共存，ignored 与 applied 互斥。
5. 为长标题、多个地域和多个标签增加响应式布局，确保桌面和移动端不重叠。
6. 为 `weekly` 增加进程锁、日志目录、失败摘要和陈旧锁恢复。
7. 手工触发一次 `FindJobsWeeklyWorkflow`，验证任务动作、工作目录、网络采集、报告生成和退出码。
8. 文档说明任务仅在登录状态运行和电池限制；如需无人值守，再单独调整任务凭据策略。

**成功标准**:

- `/jobs` 默认页面不超过 100 条，响应体低于 300 KB，本机首屏响应目标低于 500 ms。
- 所有现有筛选组合在分页后结果正确，参数在翻页后保留。
- ignored 默认不可见，标记可以添加、更新和删除，互斥状态正确。
- 桌面和移动端 Playwright 截图不存在文本溢出或控件重叠。
- Windows 定时任务至少成功实际运行一次，并留下可定位到每个来源的日志和正确退出码。
- 同时启动两次 weekly 时，第二次明确失败并指出已有运行，不损坏数据库。

**测试**:

- 分页边界、筛选组合、排序稳定性和大数据集性能测试。
- 标记新增、删除、互斥和唯一约束测试。
- Playwright 桌面与移动端视觉检查。
- 运行锁、陈旧锁、日志和计划任务命令生成测试。
- 手工 Windows Task Scheduler smoke。

**状态**: 未开始

## 阶段 5: 来源覆盖、CI 与发布验收

**目标**: 在核心准确性稳定后继续扩展厂商，并建立可持续维护门槛。

**任务**:

1. 建立 `docs/source-coverage.md`，记录目标公司、类别、官网入口、来源状态、完整度、详情质量、最近 Live smoke 和限制。
2. 将公司分为互联网平台、AI 原生公司和安全厂商三组；优先补齐当前缺失的安全厂商组，再补剩余 AI 原生公司。
3. 每批并行 2 至 3 家 adapter，每个任务独占公司文件、fixture 和测试，Codex 串行集成共享配置和注册表。
4. 每个 adapter 复用完整验收清单：官网真实性、分页、稳定 ID、详情要求、多地域、岗位类型、算法排除、职能误收、城市覆盖、薪资事实和失败重试。
5. 对大型关键词来源记录官网总量、关键词命中量和采集上限；扩充关键词前必须有漏采证据和误收回归测试。
6. 增加 GitHub Actions：Python 3.13、`uv sync`、Ruff、全量离线 pytest、迁移测试和 relevance audit fixture。
7. Live smoke 保持手工或本地定时运行，不放入普通 PR CI；输出机器可读健康报告并检测来源连续失败。
8. 更新 README、故障排查、Claude Code 状态文档，移除“仓库仍未跟踪”等过时说明；补充正式 `LICENSE`。
9. 完成全量验收后提交、推送 GitHub，并删除本文件。

**成功标准**:

- 目标公司覆盖表可以明确回答“已完成、受限、未开始”，不再以配置条目数量代替遍历完成度。
- 新 adapter 全部通过离线 fixture、Codex 真实官网 smoke 和阶段 2 相关性审计。
- GitHub Actions 在干净环境中通过，Live 健康报告能定位来源、错误和最近成功时间。
- `uv run pytest`、`uv run ruff check src tests`、迁移 smoke、Web 手工检查和计划任务 smoke 全部通过。
- README 与实际命令、模型调用方式、来源状态一致。

**测试**:

- 每个 adapter 的离线 fixture 和分页/详情失败测试。
- 来源覆盖文档与 YAML 配置一致性测试。
- GitHub Actions 干净安装测试。
- 最终全量回归、Live smoke、Web 检查、数据库备份恢复演练。

**状态**: 未开始

## 阶段门禁与提交策略

- 每个阶段开始前，Codex 给出 2 至 6 个小任务包，标明允许修改路径、禁止修改路径、测试命令和验收输出。
- 每个小任务完成后先做目标测试和 diff 审查；阶段结束时再跑全量测试。
- 核心状态、分类和推荐阶段禁止并行修改共享模块。
- Adapter 阶段允许并行开发，但只有 Codex 能修改共享配置、adapter 注册和激活状态。
- 每个阶段完成并验收后单独提交，提交信息说明为何修改，不混入本地数据库、简历、画像和生成报告。
- 任一质量门禁失败时不进入下一阶段，不通过删除测试、放宽断言或改用模拟结果绕过。

## 闭环审查记录

计划经过 4 轮闭环检查：

1. 第一轮接受并修复状态语义混用、`prune` 破坏用户标记、失败采集错误归档和 CollectRun 失败记录重复创建问题。
2. 第二轮接受并修复推荐阶段缺少岗位要求输入、简历隐私边界、Web 全量渲染和定时任务从未真实执行的问题。
3. 第三轮接受并修复迁移回滚、现有 bookmark 保留、部分来源不能用于下线判断、adapter 并行共享文件冲突问题。
4. 第四轮完整检查未发现新的可执行缺口，设计、阶段顺序、数据输入和验收标准一致。

明确延后：云部署、账号系统、自动投递、消息推送和市场薪资估算不属于本轮范围。
